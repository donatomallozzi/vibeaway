从 Vibe Coding 到 Mobility Coding

VibeAway：与 AI 对话，构建 Claude Code 及更多工具的远程接口

Donato Mallozzi Claude Code CLI · Telegram · Python 2026

我们越来越多地使用 Claude Code 或类似的 AI 开发助手。这些都是强大的工具：它们理解代码、读取文件、修改、重构、提出建议。

实际上，你很容易花上几个小时看着它们替你工作，也许一边参加线上会议，只需要偶尔介入提供新的提示词并评估结果。

但它们有一个硬性限制——需要你坐在终端前面，有时甚至是一脸茫然地盯着屏幕。😶

一天下午，我正在等 Claude 完成对一个遗留代码库的较长分析。我需要出门。无法知道它什么时候完成，无法发送后续指令，没有通知。

我合上笔记本电脑就走了，丢失了思路。
我相信任何人在类似情况下都会自然而然地想到：必须坐在电脑前才能操作，实在太可惜了——明明只需要评估结果、提供输入，哪怕远程也能做到。

那份挫败感变成了一个问题：

"我想在不在电脑前的时候也能和 Claude Code 交互。我们能不能开发一个远程接收命令的接口，既支持文字消息也支持语音？"

看似简单的技术问题——暴露一个接口——实际上远比想象中丰富。在与 Claude 的八次连续对话中，我们构建了一个我原本没有计划的东西：一个完整的系统，具备持久化会话、异步任务、语音转写、实时流式输出和安全控制。这篇文章讲述了这个故事——做出的选择、反复的思考，以及我一路走来学到的东西。

首先坦诚说明一点：已经有一些开源项目——其中一些非常流行，比如 OpenClaw——已经在解决这个问题甚至做得更多，尽管它们的重点更多在个人助手而非 vibe coding 风格的代码助手。我研究过它们。关键在于，做更多并不是我想要的。我想要的是一些简单的东西，我能逐一理解每个部分，能够控制、修改并随时关闭。在这种情况下，从零开始构建不是重新发明轮子——而是选择一个大小合适的轮子。

第一个选择：哪个渠道？

在写第一行代码之前，有一个架构决策要做：如何将提示词传送到电脑？显而易见的选项是 REST API、Web 应用或某种即时通讯服务。

我们选择了 Telegram，并非为了赶时髦。具体的技术原因是 Telegram 使用 long polling——是机器人主动调用 Telegram，而不是反过来。这意味着零网络配置：不需要端口转发，不需要开放端口，在任何企业防火墙后面都能工作。对于一个运行在家用电脑上的工具来说，这是"始终可用"和"取决于路由器设置"之间的区别。

此外，Telegram 原生支持语音消息——它将语音压缩为 OGG/Opus 格式，并通过 API 提供下载。这个功能起初看起来只是一个额外的好处，后来却在许多后续阶段变得至关重要。

初始设置简单得几乎令人尴尬：在 @BotFather 上创建一个机器人，获取一个 token，电脑就开始接收消息了。

阶段一 — 几个文件的原型

对话
👤
我想在不在电脑前的时候也能和 Claude Code 交互，支持文字和语音消息。
🤖
Telegram 非常适合做原型：long polling（无需端口转发）、原生语音消息、优秀的移动应用。10 分钟即可搭建完成。

第一个原型刻意保持最简。三个 Python 文件，一个主要依赖（python-telegram-bot），以及一个明确的目标：接收消息，传递给 claude --print，然后回复输出结果。

这类项目最大的风险是想一次做完所有事情。我忍住了。原型只做了一件事：把 Telegram 消息转化为 Claude Code 提示词。没有会话管理，没有后台任务，没有设置。只有最基本的循环。

不过有一个选择从一开始就是不可妥协的：用户白名单。Claude Code 拥有对我电脑文件系统的完全访问权限。一个没有访问控制的 Telegram 机器人就是一扇对任何知道机器人名字的人敞开的大门。Telegram 用户 ID 由 Telegram 服务器分配，客户端无法伪造——一个简单但可靠的屏障。

语音功能通过 Whisper 实现：机器人下载音频文件，转录文字，在聊天中显示转录内容，然后将文本发送给 Claude。"显示转录内容"是一个小但重要的用户体验细节：用户可以立即看到机器人听到了什么，如果 Whisper 误解了可以及时纠正。

第一个小时结束时，我已经有了一个真正能用的东西。我可以从智能手机发送消息，并收到 Claude Code 的回复。第一次端到端测试成功的那种满足感很难用语言描述。

接下来你可以想象到：我在电脑上激活了机器人服务然后出门了，继续远程改进机器人。每开发一个新功能，Claude CLI 就更新我电脑上的代码，运行测试并重启服务，使更改立即生效。

阶段二 — 会话问题

一个基本需求立即变得明显：能够重新连接到之前打开的会话并利用其上下文继续工作至关重要。

对话
👤
列表命令显示可用的会话（ID 和标题），恢复命令让我可以重新激活其中任何一个，甚至可以通过标题恢复。
🤖
我将添加 session_manager.py，它读取 ~/.claude/projects/ 中的 JSONL 文件，并通过 /proc 提取 ID、标题和打开状态。

使用了几个小时后，我遇到了下一个限制。Claude Code 将每次对话保存为磁盘上的 JSONL 文件，--continue 会恢复最后一个会话。但我在不同的项目上工作——而且我经常想恢复某个特定的对话，不一定是最后一个。

我向 Claude 提出的问题非常精确：我想能够列出会话并通过标题或部分 ID 来恢复某个会话。回答这个问题需要理解 Claude Code 如何在磁盘上组织数据——一个未文档化的目录结构——并构建一个能从中提取有用信息的读取器。

这个阶段最有趣的部分是检测当前打开的会话。Claude Code 不会留下锁文件。唯一知道某个会话是否打开的方法是查看正在运行的进程——在 Linux 上，通过检查 /proc/*/cmdline。如果命令行包含 --resume <uuid>，匹配是精确且确定的。如果没有，最近修改的会话会被标记为"可能已打开"——对不确定性保持透明，而不是假装拥有并不存在的确定性。

阶段三 — 异步转折点

既然已经走到这一步，为什么要局限于同步请求：通过 task 命令我想启动一个长时间计算，同时继续进行其他交互式会话。

对话
👤
我需要进度跟踪——我想在 Claude 完成当前会话中的长时间任务后立即收到通知。
🤖
架构变更：subprocess.run() → asyncio.create_subprocess_exec()。机器人保持响应性，任务完成时发送推送通知。

这是整个项目中最重大的一次重新思考。

最初的原型使用 subprocess.run()——本质上是阻塞的。当 Claude 在处理时，机器人是静默的：无法回复其他消息，无法接收 /cancel，什么都做不了。对于简短的响应这不明显。但对于耗时数分钟的任务来说，这是一个严重的问题。

迁移到 asyncio.create_subprocess_exec() 改变了系统的本质。机器人变成了真正的异步：你可以启动一个长时间任务，继续正常使用机器人，当 Claude 完成时收到推送通知——消息中包含完整输出。你还可以并行启动多个任务，监控它们，并取消特定的某一个。

但也需要能够中断正在运行的长时间任务，以防你意识到提示词中遗漏了重要信息。取消操作是一个精细的细节：需要同时终止操作系统进程（proc.kill()）和 asyncio 任务（asyncio_task.cancel()），并处理错误传播，确保用户即使在取消时也能收到确认通知。

回顾来看，正是这个阶段将机器人从"有用的小工具"转变为"正经的工具"。

阶段四 — 让命令有声音

对话
👤
我需要一种方式在语音消息中调用 /.. 命令，而不仅仅是作为自由提示词。
🤖
我将使用 Claude Haiku 作为分类器：转录 → 结构化 JSON → 分发到与文字命令相同的处理器。延迟约 0.5 秒。

语音消息作为自由提示词运行良好——你说些什么，Claude 就回复。但很快我想要更多：说"给我看正在运行的任务"，然后让机器人理解我是在请求列表，而不是将这句话作为提示词传递给 Claude Code。

第一反应是构建一个关键词匹配系统。"任务" → /tasks，"取消" → /cancel。容易实现，无法维护，天生脆弱。

最优雅的解决方案是使用 LLM 来分类意图——具体来说是 Claude Haiku，最快且最经济的模型。一次 API 调用，一个描述可用命令的系统提示词，模型就会返回结构化 JSON，指示要运行哪个命令及其参数。大约半秒的额外延迟，零维护成本。

让我印象深刻的是这个选择中的一个微妙的讽刺：我在用一个 AI 来解释语音命令，然后将这些命令传递给另一个 AI。但它确实有效——即使面对模糊的短语和口语化的表达也能很好地工作。

一个重要的设计细节：如果 Haiku 调用因任何原因失败（速率限制、网络错误），机器人不会阻塞。转录文本会直接作为自由提示词传递给 Claude Code。从用户角度来看完全无感，服务持续运行。

阶段五 — 等待中的沉默

到目前为止一切顺利，但一切感觉太疏离了——在助手处理过程中我没有任何反馈。

对话
👤
你之前提到了 JSON 流式输出——具体是什么意思？
🤖
claude --output-format stream-json 会实时发出 JSONL 事件。我每 N 秒编辑一次 Telegram 消息，显示部分文本加光标 ▌。

任何等待系统的设计中都存在一种张力：用户不知道是正在处理还是出了问题。一个冻结两分钟的"⏳ Claude 正在处理..."会让人焦虑。

Claude Code 支持 --output-format stream-json 标志，在处理过程中实时发出 JSONL 事件，而不是等到完成。Telegram 不支持逐字符流式输出，但允许编辑已发送的消息。解决方案是将两者结合：从 JSON 事件中累积部分文本，每两秒用当前文本和 ▌ 光标更新 Telegram 消息。

节流是必要的，以避免超过 Telegram 的速率限制（每条消息每分钟约 20 次编辑），但事实证明这也是一个用户体验的改进：每两秒更新一次是可读的，每 200 毫秒更新一次则会令人眩晕。

一个我很喜欢的小细节：部分文本末尾的 ▌ 光标。它在技术上没有什么信息量——只是说"还在写"——但它把等待从焦虑变成了期待。

技术说明 后台任务（/task）故意使用非流式模式。你等待的是最终通知——并不是在实时盯着屏幕。

阶段六 + 七 — 附件与设置：决定体验的细节

到了这个阶段，我开始需要查看一些被机器人修改或生成的文件——我需要一种方式在智能手机上打开它们。

对话
👤
一个 /sendme 命令用于以附件形式接收文件会很有用。而且我想在启动命令之前设置当前目录和流式输出频率。
🤖
两个新增功能：/sendme 支持多文件自动 ZIP 压缩 + 声明式 SETTINGS_SCHEMA 配合 /set 和 /settings。新增一个参数 = 字典里加一行。

这两个阶段源自使用中产生的实际需求。

第一个需求：我经常让 Claude 生成文件、报告或脚本——然后发现没有简便的方式从智能手机上获取它们。附件命令比预期的要丰富：单个文件直接作为附件发送，但如果使用了通配符模式（src/*.py），机器人会在内存中创建一个 ZIP 文件——不在磁盘上留下临时文件——并作为压缩包发送。自动的文件名清理是一个实用的细节：src/*.py 变成 src_py.zip，特殊字符一并处理。

第二个需求：我想在不编辑 .env 文件和重启进程的情况下配置机器人的行为。工作目录、流式输出频率、超时时间、转录语言。这里的设计选择是使用声明式 schema：添加一个新参数只需要在字典中增加一个条目，所有的验证逻辑、面板渲染和默认值重置都会自动处理。这种决策——投资于一个抽象层而不是复制粘贴——在第二次修改时就能收回成本。

阶段八 — 看不见的安全

对话
👤
让我们给转录添加安全检查：如果包含危险术语，机器人在传递给 Claude 之前先请求确认。
🤖
6 个风险类别的正则表达式模式。如果触发，我显示 Telegram 内联按钮 ✅/❌。有效载荷存储在 user_data 中，直到用户点击。

最后一个问题很微妙。文字消息是我有意识地输入的——很难意外地写出"删除所有内容"。但语音消息不同：Whisper 可能会意外转录出不正确的内容，或者我可能在嘈杂环境中说话导致不可预测的结果。考虑到 Claude Code 拥有文件系统的访问权限，一个破坏性命令的转录错误可能造成真实的损害。

解决方案是对风险模式类别进行过滤——文件删除、特权命令、意外的网络操作、任意 shell 执行、凭证访问——当检测到这些模式之一时，通过 Telegram 内联按钮发出明确的确认请求。

让我信服的设计方面是，确认在执行之前到达，并且转录内容是可见的。用户可以看到机器人理解了什么，然后决定是否继续。这不是障碍——而是透明。

有效载荷（转录文本 + 已分类的意图）被临时存储，按钮点击时检索。无需重新处理，无需额外的 API 调用。

我学到了什么

回顾这八个阶段，有些事情现在比开始时看得更清楚了。

从一个能工作的东西开始，比从一个完整的东西开始更重要。第一天的原型简单得近乎可笑。但它能用，那种直接的经验以一种任何前期规划都无法做到的方式指导了每一个后续决策。

渠道和内容同等重要。选择 Telegram 不是审美偏好——它同时解决了三个问题：身份认证、无需网络配置的传输，以及原生音频支持。选错渠道会让其他一切都变得困难得多。

最好的功能源于真实的需求——或真实的挫败感。实时流式输出源于等待时令人尴尬的沉默。会话管理源于不同项目之间的混乱。安全检查源于对语音作为一种比文字更不可控的输入方式的认识。在使用系统之前，这些需求都不是显而易见的。

用 AI 构建工具来控制另一个 AI，这有一种循环且富有成效的特质。所有代码都是在与 Claude Code 的对话中开发的，而 Claude Code 运行在我们正在扩展的同一个会话中。这其中有一种反馈循环，加速了决策并迫使你精确地表达你想要什么。

总结

在由一个单一挫败感引发的八次对话中，我们构建了一个我原本没有计划构建的东西。最终结果是一个系统，它将终端工具变得随处可用——支持文字、语音、附件、持久化会话、实时更新和合理的安全级别。

然而，最有趣的不是最终的系统。而是过程：每次迭代如何揭示下一个问题，出于实际原因做出的架构选择事后如何证明是正确的，以及与 AI 代理的协作——在我们构建工具的同一个会话中——如何改变了我思考问题的方式。

从那以后就很难停下来了。那天晚上我在沙发上继续开发。现在我的助手还具备：
