VibeAway：AIとの対話でClaude Codeのリモートインターフェースを構築する

Donato Mallozzi Claude Code CLI · Telegram · Python 2026

私たちは皆、Claude Codeや同様のAI開発アシスタントをますます活用しています。これらは強力なツールで、コードを理解し、ファイルを読み、修正し、リファクタリングし、提案してくれます。

実際には、オンライン会議に参加しながら、AIが代わりに作業する様子をただ見守るだけで何時間も過ごしてしまうことがよくあります。必要なのは時々新しいプロンプトを入力して結果を確認することだけです。

しかし、これらのツールには厳しい制約があります――ターミナルの前にいなければならない、時にはぼんやりと画面を見つめながら。😶

ある午後、Claudeがレガシーコードベースのかなり長い分析を終えるのを待っていました。外出しなければなりませんでした。いつ終わるかも分からず、フォローアップを送ることもできず、通知も届きません。

ノートPCを閉じて出かけ、作業の流れを失いました。
同じような状況にある人なら誰でも自然に思うことですが、PCの前に物理的にいなければならないのは本当にもったいない——必要なのは結果を評価してフィードバックを送ることだけで、それはリモートでもできるのにということです。

そのフラストレーションが一つの問いになりました：

「PCの前にいないときでもClaude Codeとやり取りしたい。テキストメッセージと音声の両方でリモートからコマンドを受け取るインターフェースを開発できないだろうか？」

単純な技術的問題――インターフェースを公開すること――に見えたものは、実際にははるかに奥深いものでした。Claudeとの8回の連続した対話を通じて、私は計画していなかったものを構築しました：永続的なセッション、非同期タスク、音声認識、リアルタイムストリーミング、セキュリティ制御を備えた完全なシステムです。この記事はその物語を語ります――選択、再考、そしてその過程で学んだことを。

最初に正直に断っておきます：オープンソースプロジェクトは既に存在しており、OpenClawのように非常に人気のあるものもあります。この問題に取り組んでおり、それ以上のことをしているでしょう。ただし、それらの焦点はVibe Codingスタイルのコードアシスタントというよりもパーソナルアシスタントに近いものです。それらを調べました。しかし、より多くのことをすることは私が求めていたものではありませんでした。私が欲しかったのはシンプルなもので、一つ一つの部品を理解でき、自分で制御・修正・いつでも停止できるものでした。この場合、ゼロから構築することは車輪の再発明ではなく、適切なサイズの車輪を選ぶことでした。

最初の選択：どのチャネル？

コードを一行も書く前に、アーキテクチャ上の決定を下す必要がありました：どうやってプロンプトをPCに届けるか？ 明らかな選択肢はREST API、Webアプリ、または何らかのメッセージングサービスでした。

私たちはTelegramを選びました。流行りの理由からではありません。具体的な技術的理由は、Telegramがlong pollingを使用していることです――ボットがTelegramに接続するのであって、その逆ではありません。つまり、ネットワーク設定がゼロで済みます：ポートフォワーディングも不要、ポートの開放も不要、どんな企業ファイアウォールの内側でも動作します。自宅PCで動かすツールにとって、これは「常に動く」と「ルーターの設定次第」の違いでした。

さらに、Telegramは音声メッセージをネイティブにサポートしています――OGG/Opusで圧縮し、API経由でダウンロード可能にします。最初はボーナスのように思えたこの機能が、その後の多くのフェーズで中心的な役割を果たすことになりました。

初期セットアップはほとんど恥ずかしいくらい簡単でした：@BotFatherでボットを作成し、トークンを取得すれば、PCがメッセージを受信し始めます。

フェーズ1 — 数ファイルのプロトタイプ

会話
👤
PCの前にいないときでも、テキストと音声メッセージの両方でClaude Codeとやり取りしたいです。
🤖
Telegramはプロトタイプに最適です：long polling（ポートフォワーディング不要）、ネイティブ音声メッセージ、優れたモバイルアプリ。セットアップは10分です。

最初のプロトタイプは意図的にミニマルでした。3つのPythonファイル、1つの主要な依存関係（python-telegram-bot）、そしてやるべきことの明確なアイデア：メッセージを受信し、claude --printに渡し、出力を返信すること。

この種のプロジェクトにおける最大のリスクは、一度にすべてをやろうとすることです。私はそれに抵抗しました。プロトタイプは正確に一つのことだけを行いました：Telegramのメッセージをにして、Claude Codeのプロンプトに変換すること。セッション管理なし、バックグラウンドタスクなし、設定なし。基本的なループだけです。

ただし、すぐに下した一つの選択だけは譲れませんでした：ユーザーホワイトリストです。Claude Codeは私のPCのファイルシステムに完全にアクセスできます。アクセス制御のないTelegramボットは、ボットの名前を知っている誰にでも開かれた扉です。TelegramのユーザーIDはTelegramのサーバーによって割り当てられ、クライアント側で偽装することはできません――シンプルだが堅固なバリアです。

音声はWhisperを通じて動作しました：ボットが音声ファイルをダウンロードし、文字起こしし、チャットに文字起こし結果を表示し、そのテキストをClaudeに送信します。この「文字起こしを表示する」というのは小さいけれど重要なUXの工夫でした：ユーザーはボットが何を聞き取ったかをすぐに理解でき、Whisperが誤認識した場合に修正できます。

最初の1時間の終わりには、実際に動くものが出来上がっていました。スマートフォンからメッセージを送って、Claude Codeの応答を受け取ることができたのです。あの最初のエンドツーエンドテストの満足感は言葉では表しにくいものです。

そしてその時点で想像できるでしょう：PCでボットサービスを有効にして外出し、リモートからボットの改善を続けました。新機能を追加するたびに、Claude CLIが私のPC上のコードを更新し、テストを実行し、サービスを再起動して変更をすぐに利用可能にしてくれました。

フェーズ2 — セッションの問題

根本的な側面がすぐに明らかになりました：以前開いたセッションに再接続し、そのコンテキストを活用して続行できることが不可欠でした。

会話
👤
listコマンドで利用可能なセッション（IDとタイトル）を表示し、resumeコマンドでタイトルを指定してでも任意のセッションを再開できるようにしたいです。
🤖
session_manager.pyを追加して、~/.claude/projects/からJSONLファイルを読み取り、ID、タイトル、/proc経由での起動状態を抽出します。

数時間使ってみて、次の限界に突き当たりました。Claude CodeはすべてのセッションをディスクにJSONLファイルとして保存し、--continueは最後のセッションを再開します。しかし、私は異なるプロジェクトで作業しており、必ずしも最後のセッションではなく、特定のセッションを再開したいことがよくあります。

Claudeに対して非常に具体的な質問をしました：セッションの一覧を表示し、タイトルまたは部分IDでセッションを再開したい。その回答には、Claude Codeがディスク上でデータをどのように整理しているか――ドキュメント化されていないディレクトリ構造――を理解し、そこから有用な情報を抽出するリーダーを構築する必要がありました。

このフェーズで最も興味深かったのは、現在開いているセッションの検出でした。Claude Codeはロックファイルを残しません。セッションが開いているかどうかを知る唯一の方法は、実行中のプロセスを調べることです――Linuxでは/proc/*/cmdlineを検査します。cmdlineに--resume <uuid>が含まれていれば、マッチは正確で確実です。含まれていなければ、最も最近変更されたセッションを「おそらく開いている」としてマークします――存在しない確実性を装うのではなく、不確実性に対する透明性です。

フェーズ3 — 非同期の転換点

ここまで来ると、なぜ同期リクエストに限定するのか：taskコマンドで長時間の計算を起動し、他のインタラクティブセッションを続けたい。

会話
👤
進捗追跡が必要です――現在のセッションでClaudeが長時間実行のタスクを完了したら、すぐに通知してほしいです。
🤖
アーキテクチャ変更：subprocess.run() → asyncio.create_subprocess_exec()。ボットはレスポンシブなまま、完了時にプッシュ通知を送信します。

これはプロジェクト全体で最も大きな再考でした。

元のプロトタイプはsubprocess.run()を使用していました――定義上ブロッキングです。Claudeが処理中、ボットは沈黙していました：他のメッセージに応答できず、/cancelを受け取ることもできず、何もできませんでした。短い応答では気になりませんが、数分かかるタスクでは深刻な問題です。

asyncio.create_subprocess_exec()への移行はシステムの性質を変えました。ボットは真に非同期になりました：長時間のタスクを開始し、ボットを通常通り使い続け、Claudeが完了したらプッシュ通知を受け取ることができます――メッセージには完全な出力が含まれます。複数のタスクを並行して開始し、監視し、特定のタスクをキャンセルすることもできます。

しかし、プロンプトに重要な情報を見落としたことに気づいた場合、実行中のタスクを中断できる必要もあります。キャンセルはデリケートな問題でした：OSプロセス（proc.kill()）とasyncioタスク（asyncio_task.cancel()）の両方を終了し、キャンセル時でもユーザーが確認通知を受け取れるようにエラー伝播を処理する必要がありました。

振り返ると、これはボットを「便利なウィジェット」から「本格的なツール」に変えたフェーズです。

フェーズ4 — コマンドに声を与える

会話
👤
音声メッセージで、フリープロンプトとしてだけでなく、/..コマンドを呼び出す方法が必要です。
🤖
Claude Haikuを分類器として使用します：文字起こし → 構造化JSON → テキストコマンドと同じハンドラにディスパッチ。レイテンシは約0.5秒です。

音声メッセージはフリープロンプトとしてはうまく機能していました――何かを言えば、Claudeが応答します。しかし、すぐにもっと多くのことをしたくなりました：「実行中のタスクを見せて」と言って、ボットがそれをClaude Codeに渡すプロンプトではなく、一覧のリクエストだと理解してほしかったのです。

最初の衝動は、キーワードマッチングシステムを構築することでした。「タスク」→ /tasks、「キャンセル」→ /cancel。実装は簡単、保守は不可能、本質的に脆弱です。

最もエレガントな解決策は、意図の分類にLLMを使用することでした――具体的にはClaude Haiku、最も高速で経済的なモデルです。1回のAPI呼び出し、利用可能なコマンドを記述するシステムプロンプト、そしてモデルがどのコマンドをどの引数で実行するかを示す構造化JSONで応答します。追加レイテンシは約0.5秒、メンテナンスはゼロです。

この選択で印象的だったのは、微妙な皮肉です：別のAIに渡す音声コマンドを解釈するためにAIを使っているのです。しかし、それは機能します――曖昧なフレーズや口語的な表現でもうまく機能します。

重要な設計上の詳細：Haiku呼び出しが何らかの理由（レート制限、ネットワークエラー）で失敗した場合、ボットはブロックしません。文字起こしはフリープロンプトとしてClaude Codeに直接渡されます。ユーザーの観点からは完全に透過的で、サービスの継続性が保たれます。

フェーズ5 — 待機中の沈黙

ここまでは順調でしたが、すべてが距離を感じさせるものでした――処理中にアシスタントからのフィードバックがありませんでした。

会話
👤
先ほどJSONストリーミングを提案していましたが、具体的にどういう意味ですか？
🤖
claude --output-format stream-jsonはリアルタイムでJSONLイベントを出力します。N秒ごとに部分テキスト + カーソル ▌ でTelegramメッセージを編集します。

待機システムの設計には常に緊張があります：ユーザーは何かが起きているのか、何かが壊れたのか分かりません。2分間フリーズした「⏳ Claudeが処理中...」は不安を生みます。

Claude Codeは--output-format stream-jsonフラグをサポートしており、完了を待つ代わりに処理中にJSONLイベントを出力します。Telegramは1文字ずつのストリーミングをサポートしていませんが、既に送信されたメッセージの編集は可能です。解決策は両者を組み合わせることでした：JSONイベントから部分テキストを蓄積し、2秒ごとに現在のテキストと ▌ カーソルでTelegramメッセージを更新します。

スロットリングはTelegramのレート制限（メッセージあたり1分に約20回の編集）を超えないために必要でしたが、UXの改善にもなりました：2秒ごとの更新は読みやすく、200ミリ秒ごとでは混乱を招きます。

気に入った小さなディテール：部分テキストの末尾にある ▌ カーソルです。技術的な情報量はありません――「まだ書いている」ということしか伝えていません――しかし、待機を不安から期待に変えてくれます。

技術ノート バックグラウンドタスク（/task）は意図的に非ストリーミングモードを使用します。いずれにせよ最終通知を待っているのですから――画面をリアルタイムで見ているわけではありません。

フェーズ6 + 7 — 添付ファイルと設定：差を生む細部

この時点で、ボットが修正または生成したファイルを見る必要が出てきました――スマートフォンで開く方法が必要でした。

会話
👤
添付ファイルとしてファイルを受け取る/sendmeコマンドがあると便利です。また、コマンドを実行する前に現在のディレクトリやストリーミング頻度を設定したいです。
🤖
2つの追加機能：複数ファイルの自動ZIP対応の/sendme + /setと/settingsの宣言的SETTINGS_SCHEMA。新しいパラメータ = 辞書の1行です。

これらの2つのフェーズは、使用中に現れた実用的なニーズから生まれました。

1つ目：Claudeにファイルやレポート、スクリプトの生成を依頼することがよくありましたが、スマートフォンからそれらを取得する簡単な方法がありませんでした。添付ファイルコマンドは予想以上に豊富な機能を持つことになりました：単一ファイルは直接添付として届きますが、globパターンを使用した場合（src/*.py）、ボットはメモリ内でZIPを作成し――ディスク上に一時ファイルなし――アーカイブとして送信します。自動的な名前のサニタイズは実用的なディテールでした：src/*.pyはsrc_py.zipになり、特殊文字も含めて処理されます。

2つ目：.envファイルを編集してプロセスを再起動することなく、ボットの動作を設定したいと思いました。作業ディレクトリ、ストリーミング頻度、タイムアウト、文字起こし言語。ここでの設計上の選択は宣言的スキーマを使用することでした：新しいパラメータの追加には辞書の1エントリだけが必要で、すべてのバリデーションロジック、パネルのレンダリング、デフォルトのリセットは自動的に処理されます。この種の判断――コピー＆ペーストの代わりに抽象化に投資すること――は、2回目の修正で元が取れます。

フェーズ8 — 見えないセキュリティ

会話
👤
文字起こしにセキュリティチェックを追加しましょう：危険な用語が含まれている場合、Claudeに渡す前にボットが確認を求めるようにします。
🤖
6つのリスクカテゴリに対する正規表現パターン。トリガーされた場合、Telegramのインラインボタン ✅/❌ を表示します。ペイロードはタップされるまでuser_dataに保存されます。

最後の問題は微妙なものでした。テキストメッセージは意識的に入力します――うっかり「すべて削除して」と言ってしまうことはありません。音声メッセージは違います：Whisperが予期しない文字起こしをする可能性があり、騒がしい環境で話すと予測不可能な結果になることがあります。Claude Codeがファイルシステムにアクセスできる状態で、破壊的なコマンドの文字起こしエラーは実際の被害を引き起こす可能性がありました。

解決策は、リスクのあるパターンカテゴリ――ファイル削除、特権コマンド、予期しないネットワーク操作、任意のシェル実行、認証情報へのアクセス――に対するフィルターと、これらのパターンが検出された場合にTelegramのインラインボタンによる明示的な確認リクエストでした。

私を納得させた設計上のポイントは、確認が実行前に届き、文字起こしが見える状態であることです。ユーザーはボットが理解した内容を正確に確認し、続行するかどうかを決定します。それは障害ではなく、透明性です。

ペイロード（文字起こし + 既に分類された意図）は一時的に保存され、ボタンがタップ時にそれを取得します。再処理なし、追加のAPI呼び出しなし。

学んだこと

8つのフェーズを振り返ると、最初よりも今のほうが明確に見えることがいくつかあります。

動くものから始めることは、完全なものから始めることよりも重要です。 初日のプロトタイプはほとんど滑稽なほどシンプルでした。しかし、それは動きました。そして、その直接的な経験が、事前の計画では不可能だった形でその後のすべての決定を導きました。

チャネルはコンテンツと同じくらい重要です。 Telegramの選択は美学的なものではありませんでした――3つの問題を一度に解決しました：認証、ネットワーク設定なしのトランスポート、ネイティブの音声サポート。間違ったチャネルを選んでいたら、他のすべてがはるかに困難になっていたでしょう。

最良の機能は、実際のニーズ――あるいは実際のフラストレーションから生まれます。 リアルタイムストリーミングは待機中の気まずい沈黙から生まれました。セッションは異なるプロジェクト間の混乱から生まれました。安全チェッカーは、音声がテキストよりも制御しにくい入力であるという認識から生まれました。これらのニーズは、システムを使う前には明らかではありませんでした。

AIを制御するためにAIと共に構築することには、循環的で生産的な何かがあります。 すべてのコードは、Claude Codeとの対話で開発されました。Claude Codeは、私たちが拡張していたのと同じセッションで動作していました。この中にはフィードバックループがあり、それが意思決定を加速させ、自分が何を望んでいるかについて正確であることを強制します。

まとめ

一つのフラストレーションから始まった8回の対話で、私は計画していなかったものを構築しました。その結果は、ターミナルツールを取り、テキスト、音声、添付ファイル、永続セッション、リアルタイム更新、そして合理的なレベルのセキュリティを備えて、どこからでもアクセス可能にするシステムです。

しかし、最も興味深いのは最終的なシステムではありません。それはプロセスです：各イテレーションが次の問題をどのように明らかにしたか、実用的な理由で行ったアーキテクチャの選択が振り返ると正しかったこと、そしてAIエージェントとのコラボレーション――ツールを構築していたのと同じセッションで――が、問題についての考え方をどのように変えたかということです。

その時点から、止めるのが難しくなりました。その夜、ソファからも作業を続けました。今では私のアシスタントにはこんな機能もあります：

🔊
テキスト読み上げ
プロンプトが音声だった場合、ボットの応答も音声メッセージとして届きます
🖥️
オンデマンドスクリーンショット
全画面または特定のウィンドウのスクリーンショットを送信します
📹
ウェブカメラ
PCのウェブカメラで動画を撮影して送信します

一部のユーティリティコマンドは利便性のためにハードコードされていますが、実際には適切なプロンプトがあれば、Claude Code CLIのすべての機能を活用できます。
