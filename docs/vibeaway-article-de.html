Vom Vibe Coding zum Mobility Coding

VibeAway: ein Gespräch mit einer KI, um eine Remote-Schnittstelle für Claude Code und mehr zu bauen

Donato Mallozzi Claude Code CLI · Telegram · Python 2026

Wir alle nutzen Claude Code oder ähnliche KI-Entwicklungsassistenten immer häufiger. Es sind leistungsstarke Werkzeuge: Sie verstehen Code, lesen Dateien, ändern, refaktorisieren, schlagen vor.

In der Praxis verbringt man leicht ganze Stunden damit, ihnen bei der Arbeit zuzusehen, vielleicht während man an Online-Meetings teilnimmt, und muss nur gelegentlich eingreifen, um einen neuen Prompt zu geben und das Ergebnis zu bewerten.

Aber sie haben eine harte Grenze — sie erfordern, dass man da ist, vor dem Terminal, manchmal sogar mit leerem Blick auf den Bildschirm. 😶

Eines Nachmittags wartete ich darauf, dass Claude eine ziemlich lange Analyse einer Legacy-Codebasis abschloss. Ich musste raus. Keine Möglichkeit zu wissen, wann es fertig sein würde, keine Möglichkeit, eine Nachfrage zu senden, keine Benachrichtigung.

Ich klappte den Laptop zu und ging, wobei ich den Faden verlor.
Der Gedanke, der wohl jedem in einer ähnlichen Situation kommt, ist, dass es wirklich schade ist, physisch vor dem PC sitzen zu müssen, wenn alles, was man braucht, nur das Bewerten der Ergebnisse und das Geben von Input ist, auch aus der Ferne.

Aus dieser Frustration wurde eine Frage:

"Ich möchte mit Claude Code interagieren, auch wenn ich nicht an meinem PC bin. Können wir eine Schnittstelle entwickeln, die Befehle aus der Ferne empfängt, sowohl Textnachrichten als auch Audio?"

Was wie ein einfaches technisches Problem aussah -- eine Schnittstelle bereitzustellen -- erwies sich als viel reichhaltiger. In acht aufeinanderfolgenden Gesprächen mit Claude bauten wir etwas, das ich nicht geplant hatte: ein vollständiges System mit persistenten Sessions, asynchronen Aufgaben, Sprachtranskription, Echtzeit-Streaming und Sicherheitskontrollen. Dieser Artikel erzählt die Geschichte -- die Entscheidungen, die Umdenker und was ich dabei gelernt habe.

Zunächst ein ehrlicher Hinweis: Es gibt bereits Open-Source-Projekte -- einige sehr populär, wie OpenClaw -- die dieses Problem angehen und viel mehr leisten, auch wenn ihr Fokus eher auf dem persönlichen Assistenten als auf dem Code-Assistenten im Vibe-Coding-Stil liegt. Ich habe sie mir angesehen. Der Punkt ist, dass viel mehr zu tun nicht das war, was ich suchte. Ich wollte etwas Einfaches, das ich Stück für Stück verstand, das ich kontrollieren, ändern und jederzeit abschalten konnte. In diesem Fall von Grund auf neu zu bauen war nicht, das Rad neu zu erfinden -- es war, ein Rad in der richtigen Größe zu wählen.

Die erste Entscheidung: Welcher Kanal?

Bevor auch nur eine einzige Zeile Code geschrieben wurde, gab es eine architektonische Entscheidung zu treffen: Wie gelangen die Prompts zum PC? Die naheliegenden Optionen waren eine REST-API, eine Webanwendung oder ein Messaging-Dienst.

Wir wählten Telegram, und nicht aus modischen Gründen. Der konkrete technische Grund ist, dass Telegram long polling verwendet -- der Bot ruft Telegram an, nicht umgekehrt. Das bedeutet null Netzwerkkonfiguration: kein Port-Forwarding, keine offenen Ports, funktioniert hinter jeder Unternehmens-Firewall. Für ein Tool, das auf meinem Heim-PC läuft, war das der Unterschied zwischen "funktioniert immer" und "hängt von den Router-Einstellungen ab".

Darüber hinaus verarbeitet Telegram nativ Sprachnachrichten -- es komprimiert sie in OGG/Opus und macht sie über die API herunterladbar. Diese Funktion, die zunächst wie ein Bonus wirkte, erwies sich in vielen der folgenden Phasen als zentral.

Die anfängliche Einrichtung war fast peinlich einfach: Einen Bot bei @BotFather erstellen, ein Token erhalten, und der PC beginnt, Nachrichten zu empfangen.

Phase 1 — Ein Prototyp in wenigen Dateien

Das Gespräch
👤
Ich möchte mit Claude Code interagieren, auch wenn ich nicht an meinem PC bin, sowohl mit Text- als auch mit Sprachnachrichten.
🤖
Telegram ist perfekt für einen Prototyp: long polling (kein Port-Forwarding), native Sprachnachrichten, hervorragende mobile App. Einrichtung in 10 Minuten.

Der erste Prototyp war bewusst minimal. Drei Python-Dateien, eine Hauptabhängigkeit (python-telegram-bot), und eine klare Vorstellung davon, was er tun sollte: eine Nachricht empfangen, sie an claude --print übergeben und mit der Ausgabe antworten.

Das größte Risiko bei dieser Art von Projekt ist, alles auf einmal machen zu wollen. Ich widerstand. Der Prototyp tat genau eine Sache: eine Telegram-Nachricht in einen Claude-Code-Prompt umwandeln. Kein Session-Management, keine Hintergrundaufgaben, keine Einstellungen. Nur die grundlegende Schleife.

Eine Entscheidung wurde allerdings sofort getroffen und war nicht verhandelbar: die Benutzer-Whitelist. Claude Code hat vollen Zugriff auf das Dateisystem meines PCs. Ein Telegram-Bot ohne Zugriffskontrolle ist eine offene Tür für jeden, der den Namen des Bots kennt. Die Telegram-Benutzer-ID wird von Telegrams Servern vergeben und kann clientseitig nicht gefälscht werden -- eine einfache, aber solide Barriere.

Sprache funktionierte über Whisper: Der Bot lud die Audiodatei herunter, transkribierte sie, zeigte die Transkription im Chat an und sendete den Text dann an Claude. Dieses "Transkription anzeigen" war ein kleines, aber wichtiges UX-Detail: Der Benutzer versteht sofort, was der Bot gehört hat, und kann es korrigieren, falls Whisper etwas falsch verstanden hat.

Am Ende der ersten Stunde hatte ich etwas, das tatsächlich funktionierte. Ich konnte eine Nachricht von meinem Smartphone senden und die Antwort von Claude Code empfangen. Die Zufriedenheit über diesen ersten End-to-End-Test ist schwer zu beschreiben.

Und an diesem Punkt kann man es sich schon vorstellen: Ich aktivierte den Bot-Dienst auf meinem PC und ging los, wobei ich den Bot weiterhin aus der Ferne verbesserte. Mit jeder neuen Funktion aktualisierte Claude CLI den Code auf meinem PC, führte die Tests aus und startete den Dienst neu, um die Änderungen sofort verfügbar zu machen.

Phase 2 — Das Session-Problem

Ein grundlegender Aspekt wurde sofort klar: Die Möglichkeit, sich mit einer zuvor geöffneten Session erneut zu verbinden und sie unter Nutzung ihres Kontexts fortzusetzen, war unverzichtbar.

Das Gespräch
👤
Der Listenbefehl zeigt verfügbare Sessions (ID und Titel) an, und der Fortsetzen-Befehl lässt mich jede davon wieder aktivieren, auch nach Titel.
🤖
Ich füge session_manager.py hinzu, das die JSONL-Dateien aus ~/.claude/projects/ liest und ID, Titel und Öffnungsstatus über /proc extrahiert.

Nach ein paar Stunden Nutzung stieß ich auf die nächste Grenze. Claude Code speichert jede Konversation als JSONL-Datei auf der Festplatte, und --continue setzt die letzte Session fort. Aber ich arbeite an verschiedenen Projekten -- und ich möchte oft eine bestimmte Konversation fortsetzen, nicht unbedingt die letzte.

Die Frage, die ich Claude stellte, war sehr präzise: Ich möchte Sessions auflisten und eine nach Titel oder Teilkennung fortsetzen können. Die Antwort erforderte ein Verständnis dafür, wie Claude Code Daten auf der Festplatte organisiert -- eine undokumentierte Verzeichnisstruktur -- und den Bau eines Readers, der nützliche Informationen daraus extrahierte.

Der interessanteste Teil dieser Phase war die Erkennung aktuell geöffneter Sessions. Claude Code hinterlässt keine Lock-Dateien. Die einzige Möglichkeit zu wissen, ob eine Session geöffnet ist, besteht darin, laufende Prozesse zu überprüfen -- unter Linux durch Inspektion von /proc/*/cmdline. Wenn die cmdline --resume <uuid> enthält, ist der Abgleich exakt und sicher. Falls nicht, wird die zuletzt geänderte Session als "wahrscheinlich offen" markiert -- Transparenz über Unsicherheit statt einer Gewissheit vorzutäuschen, die nicht existiert.

Phase 3 — Der asynchrone Wendepunkt

An diesem Punkt stellte sich die Frage: Warum sich auf synchrone Anfragen beschränken? Mit dem Task-Befehl möchte ich eine lange Berechnung starten und mit anderen interaktiven Sessions fortfahren.

Das Gespräch
👤
Ich brauche Fortschrittsverfolgung -- ich möchte benachrichtigt werden, sobald Claude eine lang laufende Aufgabe in einer aktuellen Session abgeschlossen hat.
🤖
Architekturänderung: subprocess.run() → asyncio.create_subprocess_exec(). Der Bot bleibt reaktionsfähig und sendet eine Push-Benachrichtigung, wenn er fertig ist.

Dies war das bedeutendste Umdenken des gesamten Projekts.

Der ursprüngliche Prototyp verwendete subprocess.run() -- per Definition blockierend. Während Claude arbeitete, war der Bot still: Er konnte nicht auf andere Nachrichten antworten, kein /cancel empfangen, nichts tun. Bei kurzen Antworten fällt das nicht auf. Bei Aufgaben, die Minuten dauern, ist es ein ernstes Problem.

Die Migration zu asyncio.create_subprocess_exec() änderte die Natur des Systems. Der Bot wurde wirklich asynchron: Man kann eine lange Aufgabe starten, den Bot normal weiternutzen und eine Push-Benachrichtigung erhalten, wenn Claude fertig ist -- mit der vollständigen Ausgabe in der Nachricht. Man kann auch mehrere Aufgaben parallel starten, sie überwachen und eine bestimmte abbrechen.

Aber es muss auch möglich sein, eine lang laufende Aufgabe abzubrechen, wenn man merkt, dass man wichtige Informationen im Prompt vergessen hat. Der Abbruch war ein heikles Detail: Sowohl der Betriebssystem-Prozess (proc.kill()) als auch die asyncio-Aufgabe (asyncio_task.cancel()) müssen beendet werden, und die Fehlerweiterleitung muss so gehandhabt werden, dass der Benutzer auch bei einem Abbruch eine Bestätigungsbenachrichtigung erhält.

Rückblickend ist dies die Phase, die den Bot von "nützlichem Widget" zu "ernsthaftem Werkzeug" transformierte.

Phase 4 — Befehlen eine Stimme geben

Das Gespräch
👤
Ich brauche eine Möglichkeit, /.. Befehle per Sprachnachricht aufzurufen, nicht nur als freie Prompts.
🤖
Ich verwende Claude Haiku als Klassifikator: transkribieren → strukturiertes JSON → Weiterleitung an denselben Handler wie der Textbefehl. ~0,5 Sekunden Latenz.

Sprachnachrichten funktionierten gut als freie Prompts -- man sagt etwas, Claude antwortet. Aber bald wollte ich mehr: "Zeig mir die laufenden Aufgaben" sagen und den Bot verstehen lassen, dass ich nach einer Liste fragte, nicht nach einem Prompt für Claude Code.

Der erste Impuls war, ein Keyword-Matching-System zu bauen. "Task" → /tasks, "Abbrechen" → /cancel. Einfach zu implementieren, unmöglich zu pflegen, per Definition fragil.

Die eleganteste Lösung war, ein LLM zur Klassifizierung der Absicht zu verwenden -- konkret Claude Haiku, das schnellste und wirtschaftlichste Modell. Ein einziger API-Aufruf, ein System-Prompt, der die verfügbaren Befehle beschreibt, und das Modell antwortet mit strukturiertem JSON, das angibt, welcher Befehl mit welchen Argumenten ausgeführt werden soll. Etwa eine halbe Sekunde zusätzliche Latenz, null Wartungsaufwand.

Was mich an dieser Entscheidung beeindruckte, ist eine subtile Ironie: Ich verwende eine KI, um Sprachbefehle zu interpretieren, die dann an eine andere KI weitergegeben werden. Aber es funktioniert -- und funktioniert gut, selbst bei mehrdeutigen Formulierungen und umgangssprachlichen Konstruktionen.

Ein wichtiges Designdetail: Wenn der Haiku-Aufruf aus irgendeinem Grund fehlschlägt (Rate-Limit, Netzwerkfehler), blockiert der Bot nicht. Die Transkription wird direkt als freier Prompt an Claude Code weitergegeben. Aus Benutzersicht völlige Stille, Dienstkontinuität.

Phase 5 — Die Stille während des Wartens

Bisher lief alles gut, aber alles fühlte sich zu losgelöst an -- ich hatte kein Feedback von meinem Assistenten während seiner Verarbeitung.

Das Gespräch
👤
Vorhin hast du JSON-Streaming vorgeschlagen -- was genau meinst du damit?
🤖
claude --output-format stream-json gibt JSONL-Events in Echtzeit aus. Ich bearbeite die Telegram-Nachricht alle N Sekunden mit dem Teiltext + Cursor ▌.

Es gibt eine Spannung im Design jedes Wartesystems: Der Benutzer weiß nicht, ob etwas passiert oder ob etwas kaputtgegangen ist. Ein "⏳ Claude verarbeitet..." das zwei Minuten lang eingefroren ist, erzeugt Angst.

Claude Code unterstützt ein --output-format stream-json-Flag, das JSONL-Events ausgibt, während es verarbeitet, statt auf den Abschluss zu warten. Telegram unterstützt kein zeichenweises Streaming, erlaubt aber das Bearbeiten einer bereits gesendeten Nachricht. Die Lösung war, beides zu kombinieren: Teiltext aus JSON-Events akkumulieren und die Telegram-Nachricht alle zwei Sekunden mit dem aktuellen Text und einem ▌-Cursor aktualisieren.

Das Throttling war notwendig, um die Rate-Limits von Telegram nicht zu überschreiten (etwa 20 Bearbeitungen pro Minute pro Nachricht), erwies sich aber auch als UX-Verbesserung: Aktualisierungen alle zwei Sekunden sind lesbar, alle 200ms wären desorientierend.

Ein kleines Detail, das mir gefiel: der ▌-Cursor am Ende des Teiltexts. Er ist technisch nicht informativ -- er sagt nur "schreibt noch" -- aber er verwandelt das Warten von Angst in Vorfreude.

Technischer Hinweis Hintergrundaufgaben (/task) verwenden bewusst den Nicht-Streaming-Modus. Man wartet ohnehin auf die endgültige Benachrichtigung -- man schaut nicht in Echtzeit auf den Bildschirm.

Phasen 6 + 7 — Anhänge und Einstellungen: Die Details, die den Unterschied machen

An diesem Punkt brauchte ich die Möglichkeit, vom Bot geänderte oder erzeugte Dateien anzusehen -- ich brauchte eine Möglichkeit, sie auf meinem Smartphone zu öffnen.

Das Gespräch
👤
Ein /sendme-Befehl zum Empfangen von Dateien als Anhänge wäre nützlich. Und ich möchte das aktuelle Verzeichnis und die Streaming-Frequenz festlegen, bevor ich Befehle starte.
🤖
Zwei Ergänzungen: /sendme mit automatischem ZIP für mehrere Dateien + deklaratives SETTINGS_SCHEMA mit /set und /settings. Ein neuer Parameter = eine Zeile im Dictionary.

Diese beiden Phasen entstanden aus praktischen Bedürfnissen, die sich aus der Nutzung ergaben.

Das erste: Ich bat Claude oft, Dateien, Berichte oder Skripte zu generieren -- und hatte dann keine einfache Möglichkeit, sie von meinem Smartphone abzurufen. Der Anhang-Befehl erwies sich als reichhaltiger als erwartet: Eine einzelne Datei kommt als direkter Anhang, aber wenn ein Glob-Muster verwendet wird (src/*.py), erstellt der Bot ein ZIP im Arbeitsspeicher -- keine temporären Dateien auf der Festplatte -- und sendet es als Archiv. Die automatische Namensbereinigung war ein praktisches Detail: src/*.py wird zu src_py.zip, Sonderzeichen inklusive.

Das zweite: Ich wollte das Verhalten des Bots konfigurieren, ohne die .env-Datei bearbeiten und den Prozess neu starten zu müssen. Arbeitsverzeichnis, Streaming-Frequenz, Timeout, Transkriptionssprache. Die Designentscheidung hier war die Verwendung eines deklarativen Schemas: Das Hinzufügen eines neuen Parameters erfordert einen einzigen Eintrag in einem Dictionary, und die gesamte Validierungslogik, Panel-Darstellung und Standardwert-Zurücksetzung wird automatisch gehandhabt. Diese Art von Entscheidung -- in eine Abstraktion zu investieren statt Copy-Paste -- zahlt sich schon bei der zweiten Änderung aus.

Phase 8 — Die unsichtbare Sicherheit

Das Gespräch
👤
Lass uns eine Sicherheitsüberprüfung für die Transkription hinzufügen: Wenn sie gefährliche Begriffe enthält, fragt der Bot vor der Weitergabe an Claude um Bestätigung.
🤖
Regex-Muster für 6 Risikokategorien. Bei Auslösung zeige ich Telegram-Inline-Buttons ✅/❌. Die Payload wird in user_data gespeichert, bis getippt wird.

Das letzte Problem war subtil. Textnachrichten tippe ich bewusst -- es ist schwer, versehentlich "lösche alles" zu sagen. Sprachnachrichten sind anders: Whisper könnte unerwartet transkribieren, oder ich könnte in einer lauten Umgebung sprechen, mit unvorhersehbaren Ergebnissen. Da Claude Code Zugriff auf das Dateisystem hat, könnte ein Transkriptionsfehler bei einem destruktiven Befehl echten Schaden anrichten.

Die Lösung war ein Filter für riskante Musterkategorien -- Dateilöschung, privilegierte Befehle, unerwartete Netzwerkoperationen, beliebige Shell-Ausführung, Zugriff auf Anmeldedaten -- und eine explizite Bestätigungsanfrage über Telegram-Inline-Buttons, wenn eines dieser Muster erkannt wird.

Der Designaspekt, der mich überzeugte, ist, dass die Bestätigung vor der Ausführung kommt, mit sichtbarer Transkription. Der Benutzer sieht genau, was der Bot verstanden hat, und entscheidet, ob er fortfahren möchte. Es ist kein Hindernis -- es ist Transparenz.

Die Payload (Transkription + bereits klassifizierte Absicht) wird temporär gespeichert, und der Button ruft sie beim Antippen ab. Keine erneute Verarbeitung, keine zusätzlichen API-Aufrufe.

Was ich gelernt habe

Wenn ich auf die acht Phasen zurückblicke, erscheinen einige Dinge jetzt klarer als zu Beginn.

Mit etwas zu beginnen, das funktioniert, ist wichtiger als mit etwas Vollständigem zu beginnen. Der Prototyp des ersten Tages war fast lächerlich einfach. Aber er funktionierte, und diese direkte Erfahrung lenkte jede folgende Entscheidung auf eine Weise, die keine Vorausplanung hätte leisten können.

Kanäle sind genauso wichtig wie Inhalte. Die Wahl von Telegram war nicht ästhetisch -- sie löste drei Probleme auf einmal: Authentifizierung, Transport ohne Netzwerkkonfiguration und native Audio-Unterstützung. Den falschen Kanal zu wählen hätte alles andere viel schwieriger gemacht.

Die besten Features entstehen aus echten Bedürfnissen -- oder echten Frustrationen 🙂 Echtzeit-Streaming entstand aus der peinlichen Stille während des Wartens. Sessions entstanden aus der Verwirrung zwischen verschiedenen Projekten. Der Sicherheits-Checker entstand aus dem Bewusstsein, dass Sprache eine weniger kontrollierte Eingabe ist als Text. Keines dieser Bedürfnisse war offensichtlich, bevor das System genutzt wurde.

Mit einer KI zu bauen, um eine KI zu steuern, hat etwas Zirkuläres und Produktives. Der gesamte Code wurde in Gesprächen mit Claude Code entwickelt, das in derselben Session lief, die wir erweiterten. Es gibt eine Rückkopplungsschleife darin, die Entscheidungen beschleunigt und einen zwingt, präzise zu formulieren, was man will.

Zusammenfassung

In acht Gesprächen, die aus einer einzigen Frustration heraus begannen, bauten wir etwas, das ich nicht zu bauen geplant hatte. Das Ergebnis ist ein System, das ein Terminal-Werkzeug nimmt und es überall zugänglich macht -- mit Text, Sprache, Anhängen, persistenten Sessions, Echtzeit-Updates und einem angemessenen Sicherheitsniveau.

Das Interessanteste ist jedoch nicht das fertige System. Es ist der Prozess: wie jede Iteration das nächste Problem offenbarte, wie architektonische Entscheidungen, die aus praktischen Gründen getroffen wurden, sich im Nachhinein als richtig erwiesen, und wie die Zusammenarbeit mit einem KI-Agenten -- in derselben Session, in der wir das Tool bauten -- die Art veränderte, wie ich über das Problem nachdachte.

Ab diesem Punkt war es schwer aufzuhören. An diesem Abend machte ich von der Couch aus weiter. Jetzt hat mein Assistent auch:
