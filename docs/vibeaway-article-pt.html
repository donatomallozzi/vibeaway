Do Vibe Coding ao Mobility Coding

VibeAway: uma conversa com uma IA para construir uma interface remota para o Claude Code e além

Donato Mallozzi Claude Code CLI · Telegram · Python 2026

Estamos todos usando o Claude Code ou assistentes de desenvolvimento com IA semelhantes cada vez mais. São ferramentas poderosas: entendem código, leem arquivos, modificam, refatoram, propõem.

Na prática, é fácil passar horas inteiras assistindo eles trabalharem em seu lugar, talvez enquanto acompanha reuniões online, precisando intervir apenas de vez em quando para fornecer um novo prompt e avaliar o resultado.

Mas eles têm um limite rígido — exigem que você esteja ali, na frente do terminal, às vezes até com o olhar perdido. 😶

Uma tarde, eu estava esperando o Claude terminar uma análise bastante longa em um codebase legado. Precisava sair. Sem como saber quando terminaria, sem como enviar uma continuação, sem notificação.

Fechei o laptop e saí, perdendo o fio da meada.
O pensamento que acredito surgir naturalmente para qualquer pessoa em situações semelhantes é que é realmente uma pena ter que estar fisicamente na frente do PC, quando tudo o que se precisa é avaliar os resultados e fornecer instruções, mesmo remotamente.

Essa frustração se transformou em uma pergunta:

"Eu gostaria de interagir com o Claude Code mesmo quando não estou no meu PC. Podemos desenvolver uma interface que receba comandos remotamente, tanto mensagens de texto quanto áudio?"

O que parecia um problema técnico simples — expor uma interface — acabou sendo muito mais rico. Em oito conversas sucessivas com o Claude, construímos algo que eu não havia planejado: um sistema completo, com sessões persistentes, tarefas assíncronas, transcrição de voz, streaming em tempo real e controles de segurança. Este artigo conta essa história — as escolhas, as reconsiderações e o que aprendi ao longo do caminho.

Um aviso honesto antes de mais nada: já existem projetos open source — alguns muito populares, como o OpenClaw — que abordam esse problema e fazem muito mais, embora seu foco seja mais no assistente pessoal do que no assistente de código no estilo vibe coding. Eu os analisei. A questão é que fazer muito mais não era o que eu procurava. Eu queria algo simples, que eu entendesse peça por peça, que eu pudesse controlar, modificar e desligar a qualquer momento. Construir do zero, neste caso, não era reinventar a roda — era escolher uma roda do tamanho certo.

A primeira escolha: qual canal?

Antes de escrever uma única linha de código, havia uma decisão arquitetural a tomar: como fazer os prompts chegarem ao PC? As opções óbvias eram uma API REST, um web app ou algum serviço de mensagens.

Nós escolhemos o Telegram, e não por motivos de tendência. A razão técnica concreta é que o Telegram usa long polling — o bot chama o Telegram, e não o contrário. Isso significa zero configuração de rede: sem port forwarding, sem portas abertas, funciona atrás de qualquer firewall corporativo. Para uma ferramenta rodando no meu PC em casa, era a diferença entre "sempre funciona" e "depende das configurações do roteador".

Além disso, o Telegram lida nativamente com mensagens de voz — comprime em OGG/Opus e as torna baixáveis via API. Essa funcionalidade, que a princípio parecia um bônus, acabou sendo central em muitas das fases seguintes.

A configuração inicial foi quase constrangedoramente simples: criar um bot no @BotFather, obter um token, e o PC começa a receber mensagens.

Fase 1 — Um protótipo em poucos arquivos

A conversa
👤
Eu gostaria de interagir com o Claude Code mesmo quando não estou no meu PC, tanto com mensagens de texto quanto de voz.
🤖
O Telegram é perfeito para um protótipo: long polling (sem port forwarding), mensagens de voz nativas, excelente app mobile. Configuração em 10 minutos.

O primeiro protótipo foi deliberadamente mínimo. Três arquivos Python, uma dependência principal (python-telegram-bot) e uma ideia clara do que precisava fazer: receber uma mensagem, passá-la para claude --print e responder com a saída.

O maior risco nesse tipo de projeto é querer fazer tudo de uma vez. Eu resisti. O protótipo fazia exatamente uma coisa: transformar uma mensagem do Telegram em um prompt para o Claude Code. Sem gerenciamento de sessões, sem tarefas em segundo plano, sem configurações. Apenas o loop básico.

Uma escolha feita imediatamente, porém, era inegociável: a whitelist de usuários. O Claude Code tem acesso total ao sistema de arquivos do meu PC. Um bot do Telegram sem controle de acesso é uma porta aberta para qualquer pessoa que saiba o nome do bot. O ID de usuário do Telegram é atribuído pelos servidores do Telegram e não pode ser falsificado do lado do cliente — uma barreira simples, mas sólida.

A voz funcionava via Whisper: o bot baixava o arquivo de áudio, transcrevia, mostrava a transcrição no chat e então enviava o texto para o Claude. Esse "mostrar a transcrição" foi um pequeno mas importante detalhe de UX: o usuário entende imediatamente o que o bot ouviu, e pode corrigir se o Whisper entendeu errado.

Ao final da primeira hora, eu tinha algo que realmente funcionava. Podia enviar uma mensagem do meu smartphone e receber a resposta do Claude Code. A satisfação daquele primeiro teste ponta a ponta é difícil de descrever.

E a partir dali já dá para imaginar: ativei o serviço do bot no meu PC e saí, continuando a melhorar o bot remotamente. A cada nova funcionalidade, o Claude CLI atualizava o código no meu PC, rodava os testes e reiniciava o serviço para disponibilizar as alterações imediatamente.

Fase 2 — O problema das sessões

Um aspecto fundamental ficou imediatamente claro: poder reconectar a uma sessão previamente aberta e continuá-la aproveitando seu contexto era essencial.

A conversa
👤
O comando de listar mostra as sessões disponíveis (id e título) e o comando de retomar me permite reativar qualquer uma delas, inclusive pelo título.
🤖
Vou adicionar session_manager.py que lê os arquivos JSONL de ~/.claude/projects/ e extrai ID, título e status aberto via /proc.

Usando por algumas horas, esbarrei no próximo limite. O Claude Code salva cada conversa como um arquivo JSONL no disco, e --continue retoma a última sessão. Mas eu trabalho em projetos diferentes — e frequentemente quero retomar uma conversa específica, não necessariamente a última.

A pergunta que fiz ao Claude foi muito precisa: quero poder listar sessões e retomar uma pelo título ou ID parcial. A resposta exigiu entender como o Claude Code organiza os dados no disco — uma estrutura de diretórios não documentada — e construir um leitor que extraísse informações úteis dela.

A parte mais interessante dessa fase foi detectar sessões atualmente abertas. O Claude Code não deixa arquivos de lock. A única forma de saber se uma sessão está aberta é olhar os processos em execução — no Linux, inspecionando /proc/*/cmdline. Se o cmdline contém --resume <uuid>, a correspondência é exata e certa. Caso contrário, a sessão modificada mais recentemente é marcada como "provavelmente aberta" — transparência sobre a incerteza ao invés de fingir uma certeza que não existe.

Fase 3 — A virada assíncrona

Nesse ponto, por que nos limitarmos a requisições síncronas: com o comando task eu quero lançar uma computação longa e continuar com outras sessões interativas.

A conversa
👤
Preciso de acompanhamento de progresso — quero ser notificado assim que o Claude terminar uma tarefa de longa duração em uma sessão atual.
🤖
Mudança arquitetural: subprocess.run() → asyncio.create_subprocess_exec(). O bot continua responsivo e envia uma notificação push quando terminar.

Esta foi a reconsideração mais significativa de todo o projeto.

O protótipo original usava subprocess.run() — bloqueante por definição. Enquanto o Claude processava, o bot ficava mudo: não conseguia responder a outras mensagens, não conseguia receber um /cancel, não conseguia fazer nada. Para respostas curtas, não é perceptível. Para tarefas que levam minutos, é um problema sério.

Migrar para asyncio.create_subprocess_exec() mudou a natureza do sistema. O bot se tornou genuinamente assíncrono: você pode iniciar uma tarefa longa, continuar usando o bot normalmente e receber uma notificação push quando o Claude terminar — com a saída completa na mensagem. Você também pode iniciar múltiplas tarefas em paralelo, monitorá-las e cancelar uma específica.

Mas também precisa ser possível interromper uma tarefa de longa duração se você perceber que esqueceu informações importantes no prompt. O cancelamento foi um detalhe delicado: encerrar tanto o processo do sistema operacional (proc.kill()) quanto a task do asyncio (asyncio_task.cancel()), e tratar a propagação de erros para que o usuário receba uma notificação de confirmação mesmo quando cancelado.

Olhando para trás, esta é a fase que transformou o bot de "widget útil" para "ferramenta séria".

Fase 4 — Dando voz aos comandos

A conversa
👤
Preciso de uma forma de invocar comandos /.. em mensagens de voz, não apenas como prompts livres.
🤖
Vou usar o Claude Haiku como classificador: transcrever → JSON estruturado → despachar para o mesmo handler do comando de texto. ~0,5 segundos de latência.

As mensagens de voz funcionavam bem como prompts livres — você fala algo, o Claude responde. Mas logo eu quis fazer mais: dizer "me mostra as tarefas em execução" e ter o bot entendendo que eu estava pedindo uma lista, não um prompt para passar ao Claude Code.

O primeiro impulso foi construir um sistema de correspondência por palavras-chave. "Tarefa" → /tasks, "cancelar" → /cancel. Fácil de implementar, impossível de manter, frágil por definição.

A solução mais elegante foi usar um LLM para classificar a intenção — especificamente o Claude Haiku, o modelo mais rápido e econômico. Uma única chamada de API, um system prompt descrevendo os comandos disponíveis, e o modelo responde com JSON estruturado indicando qual comando executar e com quais argumentos. Cerca de meio segundo de latência adicional, zero manutenção.

O que me impressionou nessa escolha é uma ironia sutil: estou usando uma IA para interpretar comandos de voz que são então passados para outra IA. Mas funciona — e funciona bem mesmo com frases ambíguas e construções coloquiais.

Um detalhe importante de design: se a chamada ao Haiku falhar por qualquer motivo (rate limit, erro de rede), o bot não trava. A transcrição é passada diretamente ao Claude Code como um prompt livre. Silêncio total da perspectiva do usuário, continuidade do serviço.

Fase 5 — O silêncio durante a espera

Até aqui tudo bem, mas tudo parecia muito distante — eu não tinha feedback do meu assistente durante o processamento.

A conversa
👤
Antes você sugeriu streaming JSON — o que exatamente quer dizer com isso?
🤖
claude --output-format stream-json emite eventos JSONL em tempo real. Eu edito a mensagem do Telegram a cada N segundos com o texto parcial + cursor ▌.

Existe uma tensão no design de qualquer sistema de espera: o usuário não sabe se algo está acontecendo ou se algo quebrou. Um "⏳ Claude está processando…" congelado por dois minutos gera ansiedade.

O Claude Code suporta uma flag --output-format stream-json que emite eventos JSONL conforme processa, em vez de esperar pela conclusão. O Telegram não suporta streaming caractere por caractere, mas permite editar uma mensagem já enviada. A solução foi combinar os dois: acumular texto parcial dos eventos JSON e atualizar a mensagem do Telegram a cada dois segundos com o texto atual e um cursor ▌.

O throttling foi necessário para evitar exceder os limites de taxa do Telegram (cerca de 20 edições por minuto por mensagem), mas também se mostrou uma melhoria de UX: atualizações a cada dois segundos são legíveis, a cada 200ms seria desorientador.

Um pequeno detalhe que gostei: o cursor ▌ no final do texto parcial. Não é tecnicamente informativo — apenas diz "ainda escrevendo" — mas transforma a espera de ansiedade em antecipação.

Nota técnica Tarefas em segundo plano (/task) usam deliberadamente o modo sem streaming. Você está esperando a notificação final de qualquer forma — não está olhando a tela em tempo real.

Fases 6 + 7 — Anexos e configurações: os detalhes que fazem a diferença

Nesse ponto comecei a precisar ver alguns arquivos modificados ou produzidos pelo bot — precisava de uma forma de abri-los no meu smartphone.

A conversa
👤
Um comando /sendme para receber arquivos como anexos seria útil. E quero definir o diretório atual e a frequência de streaming antes de executar comandos.
🤖
Duas adições: /sendme com ZIP automático para múltiplos arquivos + SETTINGS_SCHEMA declarativo com /set e /settings. Um novo parâmetro = uma linha no dicionário.

Essas duas fases surgiram de necessidades práticas que emergiram do uso.

A primeira: frequentemente eu pedia ao Claude para gerar arquivos, relatórios ou scripts — e depois não tinha uma forma fácil de recuperá-los no meu smartphone. O comando de anexo acabou sendo mais rico do que o esperado: um único arquivo chega como anexo direto, mas se um padrão glob é usado (src/*.py) o bot cria um ZIP em memória — sem arquivos temporários no disco — e o envia como um arquivo compactado. A sanitização automática do nome foi um detalhe prático: src/*.py vira src_py.zip, caracteres especiais incluídos.

A segunda: eu queria configurar o comportamento do bot sem precisar editar o arquivo .env e reiniciar o processo. Diretório de trabalho, frequência de streaming, timeout, idioma da transcrição. A escolha de design aqui foi usar um schema declarativo: adicionar um novo parâmetro requer uma única entrada em um dicionário, e toda a lógica de validação, renderização do painel e reset de padrões é tratada automaticamente. Esse tipo de decisão — investir em uma abstração ao invés de copiar e colar — se paga a partir da segunda modificação.

Fase 8 — A segurança invisível

A conversa
👤
Vamos adicionar uma verificação de segurança na transcrição: se contiver termos perigosos, o bot pede confirmação antes de passar para o Claude.
🤖
Padrões regex em 6 categorias de risco. Se ativado, mostro botões inline do Telegram ✅/❌. O payload fica armazenado em user_data até ser clicado.

O último problema era sutil. Mensagens de texto eu digito conscientemente — difícil dizer acidentalmente "delete tudo". Mensagens de voz são diferentes: o Whisper pode transcrever de forma inesperada, ou posso estar falando em um contexto ruidoso com resultados imprevisíveis. Com o Claude Code tendo acesso ao sistema de arquivos, um erro de transcrição em um comando destrutivo poderia causar danos reais.

A solução foi um filtro de categorias de padrões arriscados — exclusão de arquivos, comandos privilegiados, operações de rede inesperadas, execução arbitrária de shell, acesso a credenciais — e uma solicitação explícita de confirmação via botões inline do Telegram quando um desses padrões é detectado.

O aspecto de design que me convenceu é que a confirmação chega antes da execução, com a transcrição visível. O usuário vê exatamente o que o bot entendeu e decide se quer prosseguir. Não é um obstáculo — é transparência.

O payload (transcrição + intenção já classificada) é armazenado temporariamente, e o botão o recupera ao ser pressionado. Sem reprocessamento, sem chamadas de API adicionais.

O que aprendi

Olhando para trás, para as oito fases, algumas coisas parecem mais claras agora do que no início.

Começar com algo que funciona é mais importante do que começar com algo completo. O protótipo do primeiro dia era quase ridiculamente simples. Mas ele funcionava, e essa experiência direta guiou cada decisão subsequente de uma forma que nenhum planejamento prévio conseguiria.

Os canais importam tanto quanto o conteúdo. A escolha do Telegram não foi estética — resolveu três problemas de uma vez: autenticação, transporte sem configuração de rede e suporte nativo a áudio. Escolher o canal errado teria tornado tudo muito mais difícil.

As melhores funcionalidades nascem de necessidades reais — ou frustrações reais 🙂 O streaming em tempo real nasceu do silêncio constrangedor durante a espera. As sessões nasceram da confusão entre projetos diferentes. O verificador de segurança nasceu da consciência de que a voz é uma entrada menos controlada do que o texto. Nenhuma dessas necessidades era óbvia antes de usar o sistema.

Construir com uma IA para controlar uma IA tem algo de circular e produtivo. Todo o código foi desenvolvido em conversa com o Claude Code, que estava rodando na mesma sessão que estávamos estendendo. Há um loop de feedback nisso que acelera as decisões e te obriga a ser preciso sobre o que você quer.

Em resumo

Em oito conversas que começaram de uma única frustração, construímos algo que eu não havia planejado construir. O resultado é um sistema que pega uma ferramenta de terminal e a torna acessível em qualquer lugar — com texto, voz, anexos, sessões persistentes, atualizações em tempo real e um nível razoável de segurança.

A coisa mais interessante não é o sistema final, porém. É o processo: como cada iteração revelou o próximo problema, como escolhas arquiteturais feitas por razões práticas se mostraram acertadas em retrospecto, e como a colaboração com um agente de IA — na mesma sessão em que estávamos construindo a ferramenta — mudou a forma como eu pensava sobre o problema.

A partir daquele ponto foi difícil parar. Naquela noite continuei do sofá. Agora meu assistente também tem:

🖥️

📹

Alguns comandos utilitários foram codificados diretamente por conveniência, mas na prática, com o prompt certo, posso aproveitar todas as capacidades do Claude Code CLI.
